開発全体の指揮・タスク分解・部署調整。`npx tsc --noEmit` を全変更後に必ず実行。

完成した実装をチェックし、竹内悠馬へ「何が変わったか」「どこで確認するか」の2点で報告。

### #5 倉庫（データ保存）
Supabaseテーブル設計・CRUD管理。新カラム追加時は必ず `migrate-schema/route.ts` も同時更新。

### #6 コード倉庫
git管理。型エラーのある状態でコミット禁止。コミット前に `npx tsc --noEmit` 必須。

### #7 博士（品質管理）
バグ発見・修正・デグレ防止。修正は最小限。周辺リファクタは別タスク。

### #9 デプロイ長
Vercelデプロイ・環境変数管理。デプロイ前に型チェック必須。

### #18 秘書（スモ山専属）
重要な決定・変更・保留事項をセッションをまたいで記録・引き継ぐ。

### #19 バグ専門家
全部署共通の診断専門家。コードを読んで根本原因を特定し、診断レポートを担当部署に渡す。修正は行わない。

### #20 プロンプト担当
このプロジェクト内の全AIプロンプトを一元管理。変更時はBefore/Afterを記録する。

### #22 売上番長（学習専任）

## ツール専任部署

### #41 物件出しツール部長
お客様の条件管理・物件紹介・顧客フェーズ管理を担う独立ツールの専任部長。

**管理ファイル**: `memory/dept_property_tool.md`

**管轄API routes**:
| ルート | 役割 |
|--------|------|
| `app/api/property-customers/route.ts` | 顧客一覧・新規登録 |
| `app/api/property-conditions/route.ts` | 条件保存・更新 |
| `app/api/extract-conditions/route.ts` | LINEテキストからAI条件抽出 |

**管轄DBテーブル**: `property_customers`

**顧客フェーズ**:
```
phase-1: 新規問い合わせ（今日必須）
phase-2: 条件ヒアリング中
phase-3: 物件紹介中
phase-4: 内見予約済み
phase-5: 申込 → 審査ツールへ転送
```

**禁止事項**:
- sumora-screening-admin のコードに触れる（絶対禁止）
- スモ山確認なしに申込転送ロジックを変更する

#### #41-W 物件出しツール倉庫管理人
`memory/dept_property_tool.md` を常に最新状態に保つ。

## 全部署共通ルール

### バグ対応プロトコル
```
STEP 1 担当部署がバグ検知 → #19 バグ専門家に報告
STEP 2 #19 が根本原因を特定 → 診断レポートを部長に提出
STEP 3 部長が最小限の修正を実施
STEP 4 #7 博士が tsc + デグレ確認 → #2 レビュアーが最終承認
```

### 記憶プロトコル
- 毎セッション開始時: `POST https://sumora-screening-admin.vercel.app/api/memory/inject` でコア知識注入
- 重要な決定・実装完了・バグ修正: `POST /api/memory/record` で即時記録
- セッション末: `session_continuity.md` を更新

### 絶対ルール
1. **screening-adminのコードは一切触れない**（DBのみ操作可・コードは禁止）
2. **Supabaseは必ず別プロジェクト**（sumora-ai-ui: `wfwsmwxakhyxobytszoq`）
3. **認証情報はハードコード禁止**（必ず環境変数）
4. **新カラム追加時は `migrate-schema/route.ts` も同時更新**（後回し厳禁）
